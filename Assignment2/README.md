# Assignment 2 — Fine-tune a small open model to replace the commercial LLM

Fine-tune a Thai-first open-weight model (LoRA/QLoRA) so a **self-hosted** model can match the
Assignment-1 commercial baseline on the **same held-out eval set** — eliminating per-call API cost and
vendor lock-in at scale.

## How to run

Training is a **Colab (free T4)** notebook; data generation runs locally.

```bash
# (local) regenerate the training set — deterministic; --teacher adds LLM lexicon diversity
cd Assignment2 && uv sync
uv run python src/gen_trainset.py --teacher
```
Then open `notebooks/finetune.ipynb` in Colab (`Runtime → T4 GPU`), set `REPO_URL` (or `REPO_DIR`) and
`DRIVE_DIR`, and Run All. It does zero-shot eval → QLoRA fine-tune → fine-tuned eval (same A1 harness) →
comparison + failure analysis, and saves the adapter + reports to Google Drive.

## 1. Goal and baseline

**Goal:** match — not necessarily beat — the A1 commercial baseline with a small open model that we can
host ourselves. A1's recommendation, on the same 55-example eval set:

| A1 baseline | Amount F1 | Detail F1 | Exact | $/1k |
|-------------|-----------|-----------|-------|------|
| `google/gemini-2.5-flash-lite` | 100.0 | 98.4 | 98.2 | $0.056 |

Because the baseline is already ~98–100 F1, the win condition is **parity at near-zero marginal cost**,
not a higher F1. The **zero-shot row** (base model, no fine-tune) shows how much of that the base model
already had and how much fine-tuning actually added.

## 2. Base-model choice

**First choice — Typhoon (`scb10x/llama3.1-typhoon2-8b-instruct`)**, SCB 10X's Thai-first ~8B, for the
obvious reason: the task is Thai/mixed NER, so a Thai-first base should have the highest **zero-shot**
floor, and it continues A1 §11's "self-hosted Thai model" argument.

**What actually shipped — `unsloth/Qwen2.5-7B-Instruct-bnb-4bit` (judgment under a real constraint).**
Typhoon-8B has **no pre-quantized build**, so `load_in_4bit` must download the full ~16GB fp16 weights
and convert them — which **OOMs free Colab's ~12.7GB system RAM** (confirmed: the load crashed with
"used all available RAM" mid-download). On free hardware the practical base is a **pre-quantized 4-bit**
model: Qwen2.5-7B-4bit downloads ~5GB already-quantized, loads with a tiny RAM footprint, is multilingual
(incl. Thai), and is well-supported by Unsloth. This is exactly the cost/hardware trade-off the role asks
for: Typhoon is the better *Thai* model, but on free Colab the *runnable* model wins.

We **picked rather than ran a multi-model bake-off** — the required zero-shot eval doubles as the sanity
check, and the task is narrow enough that data quality matters more than the base. With Colab Pro / a
bigger GPU, the natural choice is `scb10x/typhoon2-qwen2.5-7b-instruct` (Thai-first, same ChatML markers)
or the 8B Typhoon; both are noted in the notebook config.

## 3. Training data

`data/train.jsonl` — **~1,000 examples**, generated by `src/gen_trainset.py`. Same 5 buckets and same
labeling rules as A1 (drop leading verbs, keep ค่า; normalize Thai numerals/number-words/`k`/`บาท`/`฿`;
amount-only & detail-only → `[]`).

| Bucket | n | How |
|--------|---|-----|
| happy_single | 250 | templated: item-lexicon × amounts × formats |
| happy_multi | 300 | templated: 2–4 singles joined by connectors/commas/newlines |
| messy | 250 | templated perturbations: Thai numerals, number-words, `k`/`บาท`/`฿`, mixed-Eng, no-space, comma, decimal |
| non_transaction | 120 | greetings/questions/number-as-time → `[]` |
| adversarial | 80 | empty/whitespace, injection, amount-only, detail-only, emoji, weird unicode, huge, injection-with-real-txn |

**Label quality:** labels are **correct by construction** — every example's surface text is built *from*
a known `(amount, detail)` pair, so labels never depend on a model guessing. Each row is additionally
self-checked against A1's `validate()` contract. An optional `--teacher` pass uses `openai/gpt-5-mini`
to *expand the lexicon* (more item names + chit-chat) for diversity; the templater still assigns labels.

**Leakage handling (no contamination):** the generator normalizes every input and **drops any row that
matches the 55-example A1 eval set** (and internal duplicates), then **asserts 0 overlap** before
writing. The notebook repeats this check. Last run filtered ~53 eval-overlaps + ~284 duplicates →
**0 overlap with the held-out eval set.**

## 4. Training setup

- **Tooling:** Unsloth (4-bit load + fast LoRA) + HuggingFace `Trainer`, **QLoRA 4-bit**. (Plain
  `Trainer` rather than TRL `SFTTrainer` — TRL's API churned against Colab's transformers 5.5; the HF
  `Trainer` path is version-stable.)
- **LoRA:** r=16, α=32, dropout=0, targets q/k/v/o + gate/up/down proj.
- **Optim:** lr 2e-4, cosine, 2 epochs, batch 2 × grad-accum 4, max_seq 1024, fp16 on T4 (bf16 on Ampere+), seed 7.
- **Loss masking:** prompt tokens set to `-100` so loss is computed only on the JSON completion.
- **Format:** chat template with A1's `SYSTEM_PROMPT` (system) + message (user) + JSON contract (assistant).
- **Hardware/cost:** Colab free **T4**, ≈ _Colab_ min, **$0**.

## 7. Trade-offs

- **Optimized for:** label correctness (construction-by-template) + strict no-leakage, and an
  apples-to-apples eval (same harness as A1).
- **Sacrificed:** templated data is cleaner and more regular than real chat; it may under-represent the
  messy long tail (handled partly by `--teacher` diversity, but not fully). Detail scoring stays strict
  (normalized exact match), so paraphrases count as wrong.
- **Small base, narrow task:** cheap to train and serve, but the model is single-purpose.

## 8. Recommendation

Ship the fine-tune **only if** it reaches commercial-baseline parity (≈ amount-F1 ≥ 99, detail-F1 ≥ 97)
**and** projected self-hosted serving cost at our volume beats the API. The decision is multi-objective:

- **Quality lift:** judged from §5 (zero-shot → fine-tuned). If zero-shot is already near parity, the
  fine-tune's value is mostly *format reliability* (always-valid JSON), not raw F1.
- **Cost:** A1's cheapest API is ~$0.056/1k; a self-hosted T4/L4 only wins past a throughput where the
  GPU stays busy. Below that break-even, **the API is cheaper and simpler** — don't self-host yet.
- **Ops:** self-hosting adds GPU/serving/monitoring burden and the ability to iterate privately on Thai
  data (no vendor lock-in, data stays in-house). Worth it at scale, not at low volume.

Honest default: at low/medium volume, **keep the A1 commercial model**; revisit the fine-tune when daily
message volume makes a GPU continuously busy.

## 9. Known limitations

- Templated training data ≠ real user-message distribution (voice-to-text noise, receipts, slang).
- No serving benchmark here (latency/throughput/cost is modeled, not measured — see bonus).
- Single eval set (55 examples) — small; CIs are wide. Larger eval needed before a real ship call.
- Thai word segmentation remains the accuracy ceiling for `detail`.
